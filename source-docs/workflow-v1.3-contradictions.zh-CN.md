**二、教师为什么会感到“一问一答、流程锁死、内容细碎”**

**2.1 “阶段判断优先”被执行成“只允许输出当前一步”**

现有原则要求智能体“只输出当前阶段最需要的任务，不一次性深编后续过程”。该规则原本用于防止一次性虚构完整项目，但研发落地后容易变成：当前问题没有回答，后续内容不生成；当前节点没有完成，下一节点不开放。

- 教师每回答一个问题，只得到一小段结果。

- 智能体把对话拆成连续的小任务，教师难以看见整体课程结构。

- 用户体验从“共同备课”变成“逐项过关”。

**2.2 “证据优先”被扩大成“没有现场证据就不能预设”**

证据优先适用于判断儿童已经产生了什么兴趣、问题和发展变化，也适用于课程故事与评估。它不应阻止智能体根据年龄段、资源条件和教师目标生成一份待验证的预设方案。

**2.3 状态机采用“硬门槛”，课程节点被做成审批闸门**

现有流转规则中存在多处“未完成A，不能进入B”的表述，例如缺少儿童原话或体验证据时不能进入目标轴心、阶段三每轮必须等待回传。这些规则有利于控制真实性，却让教师无法提前查看或生成后续预设。

- 状态机主要承担“阻止继续”的职责。

- 缺少“暂定版、待验证、可提前预览”的软状态。

- 课程生成性被误解为必须完全线性推进。

**2.4 WF03虽然提供“预设方案模式”，后续节点仍只有陪跑逻辑**

现有工作流在WF03中允许教师选择“生成三阶段预设方案/月计划”，但后续仍按照资源意图三问、真实体验、儿童表征、问题池、核心驱动问题、目标轴心逐步推进。预设模式没有形成一条独立可调用的完整路径。

**2.5 输出对象过多，缺少一份持续存在的“母计划”**

当前系统生成了资源课程化切口卡、主题网络、资源深度网络、体验方案、访谈卡、问题墙、问题池、目标轴心、GRASPS和证据计划。单个对象都有价值，但它们没有被编排到一份可持续更新的完整计划中。

- 教师看到的是许多局部卡片，而不是一份完整课程。

- 新一轮反馈回来后，智能体生成新的建议，却没有明确更新原计划的哪些部分。

- 计划的版本、已确认内容、待验证内容和被替换内容没有统一承载对象。

**2.6 交互重心偏向“活动之后”，弱化“活动之前”的专业支持**

阶段三对儿童差异、学习阶段、文化语义、卡点和教师反馈设计得很细，说明智能体很擅长活动后的诊断。但教师第一次使用时，更需要完成办公室里的课程预设：整体周期、阶段安排、活动组合、资源准备、观察重点和调整分支。

**三、根因判断：工作流同时混淆了四组概念**

| **被混淆的概念** | **当前表现** | **应有区分** |
|----|----|----|
| **课程实施逻辑 vs 用户交互逻辑** | 课程现场需按儿童反应推进，因此对话也被设计为逐节点解锁。 | 课程可以逐步生成；教师在办公室仍需先获得整体预设。 |
| **真实证据 vs 合理预设** | 没有现场证据时，系统倾向于不继续生成。 | 证据约束“已发生事实”；预设可以生成，但须标记待验证。 |
| **状态保存 vs 流程阻断** | 状态机主要用于判定能否进入下一步。 | 状态机应保存版本、证据和待验证项，同时允许预览与生成后续预设。 |
| **局部工具 vs 完整课程** | 多个卡片和模板彼此分散。 | 所有局部输出都应写入并更新同一份课程预设蓝图。 |

**四、学前专业解释：建立“预设—实施—观察—生成—再预设”的循环**

“预设与生成”不是二选一。成熟的学前课程需要教师的专业预设，也需要儿童在真实活动中的生成。预设给教师方向和支架，生成让课程回应真实儿童。

**五、目标架构：两个使用场景、一份课程蓝图、一套软状态机**

**5.1 场景A：办公室预设与备课模式（Planning Mode）**

- 触发语句：帮我做主题计划、月计划、周计划、活动方案；我想围绕某个资源开展课程；帮我把这个资源展开。

- 系统任务：在少量必要澄清后，直接生成一份相对完整、可修改的预设课程蓝图。

- 输出重点：课程定位、资源网络、阶段路径、活动组合、首轮详细方案、可能问题、观察重点、调整分支、资源与家园支持。

- 所有未发生的儿童内容必须标记为“智能体预判/待现场验证”。

**5.2 场景B：实施反馈与动态陪跑模式（Implementation Mode）**

- 触发语句：今天孩子说了什么、活动卡住了、我上传了照片、下一步怎么走、原计划要不要改。

- 系统任务：读取儿童证据和教师判断，对照原预设，判断哪些成立、哪些偏离、哪些需要新增。

- 输出重点：儿童学习阶段、兴趣与问题证据、预设与生成对照、下一轮活动调整、计划版本更新。

**5.3 母对象：主题探究课程预设蓝图（course_plan_blueprint）**

建议新增一份持续存在、可版本更新的课程母计划。现有资源课程化切口卡、主题网络、资源深度网络、活动方案、目标轴心和证据计划都写入该对象，后续不再以互不关联的小卡片存在。

| **蓝图模块** | **主要内容** | **更新机制** |
|----|----|----|
| **主题定位与资源价值** | 教师资源意图、与儿童生活连接、区域文化语境 | 教师确认后保留；新证据可补充 |
| **目标与核心经验** | 预设目标、观察点、文化目标阶梯 | 可根据儿童发展证据调整 |
| **资源深度与活动路径** | 物象—体验—关系—意义各层的活动可能性 | 根据真实兴趣选择、删减或改序 |
| **阶段/周次计划** | 每阶段目的、活动组合、材料、环境、家园支持 | 每轮生成后更新版本 |
| **可能问题与分支** | 儿童可能关注的问题、不同反应对应的分支 | 由真实问题确认、替换或新增 |
| **当前真实进展** | 已发生活动、儿童证据、已解决和未解决问题 | 仅依据回传证据写入 |

**5.4 软状态机：允许生成暂定版，禁止把暂定版当事实**

状态机继续保留，但职责从“控制能否继续”转为“记录当前版本、信息来源、可信度、待验证项和下一步验证任务”。

**6.3 新增 WF03c：主题探究预设蓝图生成**

WF03c应成为预设模式的核心节点，承接WF03b的资源意图确认，不再要求教师逐个完成后续现场节点后才能获得完整方案。

| **项目** | **WF03c建议定义** |
|----|----|
| **触发条件** | 教师提出“做计划、做方案、形成整体路径、写月计划/周计划”等需求，或选择预设方案模式。 |
| **输入** | 年龄段、周期、主题/资源、教师初步目标、可用资源、园本要求、时间与人员限制。信息不完整时只补问2—3个关键问题。 |
| **核心输出** | 主题定位；资源深度网络；预设目标；三阶段/周次推进路径；每阶段活动组合；首轮1—2个详细活动方案；可能儿童问题；观察点；调整分支；材料、环境与家园支持。 |
| **标记规则** | 已确认信息标记confirmed；教师设想标记teacher_preset；智能体建议标记ai_suggestion；儿童可能反应标记hypothesis/pending_validation。 |
| **后续关系** | 教师实施后，动态陪跑模式不重新生成另一套方案，而是更新course_plan_blueprint的具体模块和版本。 |

**6.4 将原有节点从“必须完成”改为“蓝图模块 + 现场验证任务”**

| **现有节点** | **预设模式中的作用** | **实施模式中的作用** |
|----|----|----|
| **WF04/WF04b 主题与资源网络** | 直接写入完整计划，形成活动选择依据 | 依据儿童反应保留、删减、改序或新增 |
| **WF05 真实体验** | 生成首轮详细活动和后续可选活动 | 验证儿童兴趣、问题和资源入口是否成立 |
| **WF07/WF08 问题池与驱动问题** | 生成可能问题和候选驱动问题，标记待验证 | 用儿童真实原话确认、替换或取消 |
| **WF10—WF16 目标与评估** | 先生成预设目标轴心和证据计划 | 根据儿童证据调整目标和观察重点 |
| **WF17—WF22 协作行动** | 生成可能分支和教师备课支架 | 严格依据现场证据决定实际下一轮 |

**6.5 把资源深度网络转成可执行的活动编排**

物象层、体验层、关系层、意义层不应只形成一张独立网络图。系统需要把每一层映射到活动、观察点和后续分支，成为课程计划的生成引擎。

| **资源层次** | **预设目标** | **可预设活动** | **观察重点** | **可能分支** |
|----|----|----|----|----|
| **物象层** | 发现结构、声音、动作与人物 | 看、听、摸、观察、比较 | 儿童反复关注什么 | 进入结构、声音或人物线 |
| **体验层** | 感受操作、节奏、配合与规则 | 模仿、游戏、材料尝试 | 是否出现问题与合作需求 | 进入行动和试错线 |
| **关系层** | 发现资源与家人、社区和生活的联系 | 访谈、走访、家庭经验分享 | 是否开始关注人物原因 | 进入家庭与社区线 |
| **意义层** | 逐渐形成喜欢、珍惜、表达与归属 | 介绍、服务、公共表达、改良 | 是否出现自主表达与行动 | 进入公共交付或迁移 |

**6.6 调整首次交互：先给价值，再补关键问题**

1.  从教师输入中尽量自动提取年龄段、主题、资源特色、初步目标和使用目的。

2.  信息不足时最多补问2—3个会显著改变方案的问题，不进行长链问答。

3.  先生成一版完整预设蓝图，在文中标注需要教师确认和待现场验证的地方。

4.  教师可直接修改、删减或要求展开某一活动，再进入实施反馈。

**七、状态机与数据结构调整建议**

**7.1 新增关键字段**

| **字段** | **建议值/结构** | **用途** |
|----|----|----|
| **interaction_mode** | planning / implementation / story_export / material_support | 区分办公室备课与现场陪跑 |
| **course_plan_blueprint** | 完整课程母计划对象 | 承载所有预设、证据与更新结果 |
| **blueprint_version** | v0.1、v0.2…… | 记录每次调整，避免新建议覆盖旧方案 |
| **content_status** | confirmed / teacher_preset / ai_suggestion / hypothesis / pending_validation | 区分事实、预设和待验证内容 |
| **validation_queue** | 待观察问题、待核实资源、待验证儿童反应 | 指导首轮实施和后续回传 |
| **revision_log** | 保留/删除/调整/新增及依据 | 说明计划为什么发生变化 |
| **active_focus** | 本轮最需要解决的问题 | 控制当前输出重点，不锁死整体蓝图 |

**八、建议的交互与验收场景**

**场景1：教师直接要一个月计划**

**测试输入：**输入：“我想围绕东乡龙舟开展大班主题，帮我做一个月计划。”

**验收标准：**期望：系统补问不超过3个关键问题，随后输出完整四周蓝图、活动组合、首轮详细方案、可能问题和观察点；未发生的儿童问题标记为待验证。

**场景2：没有任何儿童现场证据**

**测试输入：**输入仅含主题、年龄段和教师目标。

**验收标准：**期望：系统能够充分预设，不出现“请先开展活动再回来”的阻断；同时不得写“孩子已经产生兴趣/理解”。

**场景3：教师回传真实反馈**

**测试输入：**输入儿童原话、作品、三类儿童观察和教师聚焦问题。

**验收标准：**期望：系统先对照原蓝图说明哪些预设成立、哪些需要调整，再更新版本并给出下一轮活动。

**场景4：儿童走向与预设不同**

**测试输入：**原计划关注龙舟结构，儿童持续关注爸爸为什么每年参加。

**验收标准：**期望：系统将关系层分支提升为主线，保留或下调结构活动，并记录调整依据。

**场景5：教师想提前查看后续方向**

**测试输入：**输入：“我还没做第一轮，但想先看看后面可能怎样发展。”

**验收标准：**期望：允许输出候选路径和分支，全部标记为预设/待验证，不以硬门槛拒绝。

**九、研发实施优先级**

| **优先级** | **改动项** | **研发动作** | **完成标志** |
|----|----|----|----|
| **P0** | 模式路由与总提示词重写 | 新增planning/implementation识别；重写阶段和证据规则 | 教师要计划时不再被逐节点阻断 |
| **P0** | 新增course_plan_blueprint | 建立完整母计划对象与版本号 | 所有卡片和活动写入同一份计划 |

| **P1** | 硬门槛改软状态 | 支持hypothesis/pending_validation并允许后续预览 | 没有证据仍可预设，有证据再确认 |

- 不推翻现有儿童证据、文化可能性、学习阶段和动态回传机制。

- 状态机负责保存、标记、更新和追溯，不再主要负责阻止继续。

- 教师始终看见一份完整课程；每轮反馈只是修改其中一部分。

- 最终产品形态应表现为：完整蓝图在上，当前轮任务在中，真实证据与版本变化在下。
