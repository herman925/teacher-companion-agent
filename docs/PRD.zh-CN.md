## 1. 问题陈述

番禺的幼儿园教师需要围绕本土文化资源（龙舟、趁墟、祠堂、醒狮……）开展**主题探究/项目化探究课程**。要做好这件事，需要一个高要求的循环：先想清楚为什么选这个资源，把它转译成儿童能真实进入的切口，收集儿童真实问题，推导核心驱动问题，建立目标与评估轴心，围绕儿童现场反应开展多轮行动循环，最后沉淀出有证据支撑的**课程故事**。

大多数教师很难独自撑起这个循环。而通用的`聊天机器人`会以两种相反的方式辜负她们：

1. **编造** —— 普通大模型会顺手虚构儿童的兴趣、问题和课程进展，产出精美但虚假、毫无证据价值的课程文档。
2. **表单式僵硬** —— 如果照本宣科地实现 V1.3 工作流规范（它本身以表格、模板和状态机的形式书写），智能体就会变成一个官僚化的向导：先逐项盘问字段，才肯给出任何有用的东西。教师的体验是*在给机器人填表*，而不是被一位同行陪跑。

产品的核心命题：**严格程度要达到规范的要求——但把严格对准模型，而不是对准教师。**

## 2. 产品愿景

一个基于网页的**陪跑式对话平台**：教师自然地聊自己的班级和资源，智能体则：

- **识别教师所处的入口**（五种模式：从零陪跑 / 已有主题优化 / 过程中续聊 / 课程故事整理 / 素材支持），而不是让教师自己申报；
- **动态提问** —— 每次只问一个聚焦的问题，永远附带具体示例答案，凡是能从对话或已存状态推断出来的信息绝不重复追问；
- **拒绝编造** —— 没有儿童证据时，只给"可能方向"加一个下一步补证据任务；
- **每轮固定输出闭环**：本轮可以去做什么 → 建议生成/使用哪些素材 → 回来请告诉我什么 → 我收到后会继续帮你做什么；
- **等待真实** —— 行动轮次之间，智能体暂停等待教师带着真实现场反馈回来，再调整路径；
- **有记忆** —— 账号、每门课程的 `course_state`、聊天历史，以及（后续）向量化的教师画像，让第 N+1 轮比第 N 轮更懂这个班。

V1.3 规范是**行为契约**；平台的职责是把它隐形地兑现。教师感受到的应该是一位懂行的同事，状态机则完全留在后台。

## 4. 核心交互命题：动态识别，而非表单填写

这是本 PRD 的灵魂章节。V1.3 规范本身已经暗含这个方向（WF03b 的三问必须每问带例子；WF20d 要求智能体在请教师填写第三句聚焦反馈之前，先结合上下文给出 2–3 个示例）。我们把它推广为全平台的交互契约：

### 4.1 动态识别契约（借鉴"追问式访谈"方法论）

1. **先读后问。** 每一轮先读取 `course_state` 与对话历史。已知或可推断的信息，*永远不再问*。（对应规范原则：状态机优先。）
2. **一次一问。** 建档/引导阶段每轮最多提一个聚焦问题，并用一句话说明*为什么问*。
3. **必附推荐答案。** 每个问题都配 2–3 个贴合上下文、可直接点选或改写的示例答案——绝不给空白栏。（是 WF03b"每问带例子"与 WF20d"先给示例"的全面推广。）
4. **接受杂乱，提取结构。** 教师可以用大白话、语音转文字、问题墙照片来回答。由智能体从杂乱输入中提取状态字段；教师永远不需要面对字段表。
5. **入口靠推断，确认要轻。** WF01 的入口识别是*智能体做的分类*，以一句轻确认呈现（"听起来你已经带着孩子做过一轮了，我们接着上次的进度聊？"），而不是让教师看菜单做选择题。
6. **已满足的关卡静默跳过。** 如果教师第一条消息已包含资源意图和班级信息，智能体直接进入切口卡草稿。阶段关卡是*被检查的*，不是*被表演的*。
7. **处处留出口。** "先跳过 / 我不确定 / 你先给个初稿"永远是合法回答；智能体标注"待现场确认"后继续推进（正是 WF03b 的规定）。

### 4.2 严格的部分（对准模型）

运行时护栏（§7）逐轮强制执行：

- **证据优先**：`children_evidence` 中没有记录时，不得输出关于儿童发现/理解/感受的断言；违例被拦截并重新生成。
- **文化留在后台**：文化线索只作为面向教师的后台提示，且每条必须转译为儿童可行动的小任务；把"传承精神/文化责任"当作儿童目标属于校验失败。
- **闭环必须在场**：每个收尾消息必须包含四要素闭环（做什么/用什么素材/回来告诉我什么/我会继续帮什么）。
- **阶段关卡合法性**：没有儿童证据不得进入阶段2（目标轴心）；关卡未满足不得输出长篇方案；`awaiting_feedback` 状态下不得虚构循环进展。
- **禁止全员项目化**：任何项目级方案之前必须先做主题适配性筛查；短活动型主题只给轻量方案。

## 5. 范围

### 5.1 本期范围

1. **双语 PRD**（本文档 + PRD.md）。
2. **开发护栏（harness）** —— 从化龙平台护栏改造的仓库治理（门禁、术语一致性、结构检查），加上本项目特有检查（course_state 模式校验、提示词规则检查）。
3. **架构与 API 调研** —— MiniMax（默认）、GLM 与 Kimi（备选）、Qwen（探索）的文档化评估；腾讯云开发（CloudBase）作为后端首选方案。
4. **最小闭环网页演示**（规范§7）：资源输入 → WF03b 三问（对话式）→ 资源课程化切口卡 → 适配性筛查 → 一轮真实体验任务 + 访谈卡 → 教师回传（含三类儿童观察 + 三句聚焦反馈）→ 儿童问题池 + 文化可能性提示 → 候选核心驱动问题 → 一轮协作行动 → 课程故事片段。HTML + TypeScript，GSAP 动效，运行时护栏生效，模型适配器可插拔。
5. **DESIGN.md** —— 演示界面的语义化设计系统。

### 5.2 暂不做（延后）

- **AI 绘图/图像生成**（环创图式、海报配图）——现阶段只提供文字支架。
- 微信小程序封装（先网页；试点有需求再做）。
- 演示中的真实账号系统（演示用本地存储；CloudBase 认证只做设计不做实现）。
- 多教师协作同一课程；管理后台。
- 儿童证据的照片/音频自动分析（教师描述为主；上传仅存储）。
- 模型微调；只做提示词侧工作。

## 6. 用户故事

**建档与识别**
1. 作为教师，我希望开口只说"我想带孩子做龙舟"，剩下该问什么由智能体来定，这样我不用先填一份登记表才能得到帮助。
2. 作为课程进行中的教师，我希望智能体认得我、接着我的课程状态继续，这样我永远不用重新介绍我的班。
3. 作为答不上"为什么想做这个资源"的教师，我希望有可选的候选意图和"待现场确认"选项，这样不确定不会卡住我。
4. 作为教师，我希望每个问题都带可点选的示例答案，这样在手机上几秒钟就能回答。

**阶段0–1（意图、适配、体验、问题）**
5. 作为教师，我希望根据我的回答生成资源课程化切口卡，让我看到儿童入口而不是成人知识点。
6. 作为教师，我希望得到诚实的适配性筛查（短活动型/主题探究型/项目化潜力型），这样我不会把主题撑不起的项目硬做大。
7. 作为教师，我希望第一轮体验方案带观察重点和访谈卡（幼儿3问 + 成人3问），这样第一次外出就能带回可用证据。
8. 作为教师，我希望活动后把儿童原话/照片一股脑发进来，拿回一份分好类、剔除了成人化表述的问题池，让真实儿童问题驱动课程。
9. 作为教师，我希望文化可能性提示以*后台备注 + 儿童小任务*的形式挂在儿童问题上，这样文化做深而不喊口号。

**阶段2–3（目标、循环）**
10. 作为教师，我希望得到 2–3 个按儿童性/真实性/行动性/公共性/文化可能性校验过的候选核心驱动问题，这样我是在判断中选择，而不是从零起草。
11. 作为教师，我希望目标轴心（核心理解、四维目标、文化目标阶梯、GRASPS）以待确认草稿的形式生成，这样评估思维不需要先上一场工作坊。
12. 作为教师，我希望每轮行动都以输出闭环收尾，这样我永远知道要做什么、要带什么回来。
13. 作为教师，我希望智能体*等*我的真实反馈，而不是替我写第二轮的虚构剧情，这样记录保持真实。
14. 作为教师，我希望回传三类儿童观察和三句聚焦反馈时，第三句先有智能体结合上下文拟的示例，这样聚焦反思只花三分钟。
15. 作为教师，我希望智能体在分析时*先回应我提出的待判断问题*，这样我感到被倾听，而不是被流程处理。
16. 作为教师，我希望卡壳点被转化为下一轮探究任务，这样失败变成课程。
17. 作为教师，我希望只有当项目化信号累计≥3个时才收到项目化探究提醒，这样规模是从证据里长出来的。

**阶段4–5（交付、故事）**
18. 作为教师，我希望成果展示的准备让孩子用自己的话和作品来表达，这样展示不是背稿汇演。
19. 作为教师，我希望导出课程故事前先做材料缺口检查，这样缺口被低成本补上而不是被文字糊上。
20. 作为教师，我希望课程故事从真实的循环记录中按真实发生顺序提炼，这样它读起来是真实实践，而不是倒写出来的完美设计。
21. 作为教师/园长，我希望多版本导出（完整案例/汇报摘要/申报/公众号），一门课服务多个场合。

**平台**
22. 作为教师，我希望有账号、有聊天与课程历史，这样陪跑越久越懂我的班。
23. 作为平台运营者，我希望每次模型调用都过护栏校验并留下违例日志，这样提示词退化看得见。
24. 作为平台运营者，我希望模型适配层与供应商无关（MiniMax 默认；GLM/Kimi 备选；Qwen 开关控制），这样换模型是运维决策而不是重写代码。
25. 作为平台运营者，我希望 course_state 每轮存版本化快照，这样状态损坏可恢复、可审计。

## 7. 系统需求

### 7.1 状态机（源自规范§2，规范性要求）

- `course_state` 字段：`course_id`、`teacher_mode`、`class_profile`、`theme_resource`、`teacher_resource_intent`、`resource_entry_card`、`theme_fit_level`、`children_evidence`、`child_question_pool`、`driving_question`、`goals_assessment_axis`、`cycle_history`、`child_learning_stage`、`project_signal_level`、`story_materials`、`child_participation_difference`、`teacher_focus_feedback`——外加平台字段 `stage`、`completed_nodes`、`awaiting_feedback`、`pending_confirmations`、`schema_version`。
- 阶段流转遵循规范§2的流转规则（如：无儿童证据不得进入阶段2；阶段5导出前先做缺口检查）。流转由**确定性引擎**计算，不由大模型决定：大模型提议状态增量（state delta），引擎校验后应用。
- `course_state` 的 JSON Schema 存放于 `harness/schema/course-state.schema.json`，是演示、文档与提示词模板的唯一事实来源。

### 7.2 运行时护栏（模型治理）

分层防御，成本从低到高：

| 层 | 机制 | 拦截对象 |
|---|---|---|
| L1 系统提示词 | 规范§1原则 + §6禁止规则编译进系统提示词；按阶段加载提示词模块（规范§4循环提示词） | 绝大多数违例，从源头避免 |
| L2 结构化输出 | 模型必须以 JSON 模式/工具调用返回 `{reply_markdown, state_delta, closure_loop, evidence_refs}` | 缺失闭环、状态格式错误 |
| L3 确定性校验器 | 生成后检查：闭环四要素完整性；`state_delta` 的阶段关卡合法性；编造启发式（无 `evidence_refs` 的儿童断言）；儿童任务中的成人口号词表扫描 | 穿透 L1/L2 的规则违例 |
| L4 带反馈重生成 | L3 失败：注入违例报告重试一次；再失败：降级为安全模板 + 记录日志 | 顽固违例 |

违例日志是一等公民产物（对应 P3 的可观测性诉求）。

### 7.3 模型适配层

- OpenAI 兼容的 chat-completions 客户端；供应商注册表：**MiniMax（默认）→ GLM → Kimi** 故障转移链；**Qwen** 置于评估开关之后。
- 各供应商需满足：SSE 流式输出、JSON/工具调用模式、中文系统提示词遵循度。调研结论记录在 [MODEL-APIS.md](./MODEL-APIS.md)。
- 演示版：API 密钥由用户在前端输入（localStorage 存储，绝不入库入仓）。生产版：密钥只存在于 CloudBase 云函数中。

### 7.4 后端（设计目标：腾讯云开发 CloudBase）

- 网页 SPA（静态托管）+ CloudBase 认证（手机号/微信）+ 文档型数据库集合：`users`、`courses`（course_state + 快照）、`messages`、`materials`、`violations`。
- 云函数作为大模型代理（密钥服务端保管、SSE 透传），护栏校验器在同一云函数内运行。
- **个性化路线图**：v1 —— 结构化画像（地区、年龄段、班额、资源库、风格偏好）注入提示词；v2 —— 对教师课程/聊天语料做向量化检索增强（候选：腾讯向量数据库或库内向量字段）；全量聊天历史向量化明确*放到更后面*。
- 细节与能力核实见 [ARCHITECTURE.md](./ARCHITECTURE.md)。

### 7.5 非功能需求

- **语言**：产品界面与智能体输出为简体中文；代码与工程文档以英文为主、双语对照。
- **延迟**：MiniMax 默认模型下首个流式 token p50 < 3秒；闭环校验额外开销 < 150ms。
- **真实性**：经校验的输出中编造儿童证据零容忍（L3 拦截；以违例日志度量）。
- **合规**：数据境内存储；只使用已备案模型；未成年人数据最小化——平台存储的是教师的观察记录，上传由教师中介；PIPL 相关说明见 ARCHITECTURE.md。
- **成本上限（试点）**：约 50 名教师，默认模型月度花费目标 < ¥500。

## 8. 演示验收标准（本期）

1. 打开静态页面，输入 API 密钥，键入"我想带中班孩子做醒狮"，即得到：入口识别（静默完成）、带示例答案的 WF03b 对话、渲染出的切口卡——**全程没有任何表单界面**。
2. 拒绝回答（"不知道，你帮我想"）仍能产出带"待现场确认"标记的切口卡。
3. 在证据存在之前索要完整项目方案会被*礼貌拒绝*并给出下一步补证据任务（L3 关卡可见地触发——在调试抽屉中展示）。
4. 提交模拟教师回传（原话 + 卡点 + 三类观察 + 三句聚焦）后循环推进；回复*先回应*教师的第三句聚焦问题。
5. 每个收尾消息都以独立视觉组件渲染四要素闭环。
6. 调试抽屉展示：当前阶段、逐轮状态差异、关卡检查通过/失败、当前供应商。（严格性对我们可见，对教师隐形。）
7. 通过配置将供应商从 MiniMax 切到 GLM，无需改代码。
8. 页面经真实浏览器验证（遵循仓库验证规范），动效使用 GSAP，设计符合 demo/DESIGN.md。

## 9. 度量指标（试点）

- **反表单化**：产出第一个有价值产物（切口卡）前的提问中位数 ≤ 4；建档问题中通过点选示例回答的比例 ≥ 50%。
- **真实性**：每百轮 L3 违例率（提示词迭代后目标 < 2）；编造类违例流出到教师端 = 0。
- **陪跑性**：完成 ≥2 次完整回传往返的课程占已开课程比例 ≥ 60%（这是产品的立身之本）。
- **故事价值**：阶段5导出物中，教研员无需结构性返工即可使用的比例。
